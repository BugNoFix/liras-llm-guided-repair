================================================================================
  FIGURE EXPLAINERS
  Cross-Comparative Study on LLM-Guided DSL Code Repair
================================================================================

Study context
-------------
This study evaluates automated DSL (Domain-Specific Language) code repair
guided by large language models (LLMs).  A generator produces DSL code from a
natural-language scenario, a local compiler validates it, and — when compilation
fails — the compiler output is fed back to the model for iterative repair (up
to 10 iterations).

The experiment crosses three independent factors to form 8 configurations
(C1–C8):

  Factor 1 – Model:            gemini-2.5-flash  |  gemini-3-pro-preview
  Factor 2 – Repair Prompt:    SPR1.txt           |  SPR2.txt
  Factor 3 – Few-Shot Examples: No                |  Yes

Each configuration is run across 4 scenarios (Scenario_06, _011, _016, _029)
× 5 generative system prompts (SP1–SP5), giving 20 runs per config and
480 runs total (3 015 iteration rows).

Data source: Report/Histories/combined_run_histories.csv
    Produced by: Utils/compile_run_histories.py  (merges per-config CSVs
    created by Utils/collect_run_history.py from raw run_metadata.json files).


================================================================================
  Figure 1 — Success Rate per Configuration (95 % Wilson CI)
  File: Report/Figures/fig01_success_rate_ci.png
================================================================================

Metric
------
Success rate = (number of runs reaching compiler_error_score == 0) / (total
runs) for each configuration C1–C8.

How the data is produced
------------------------
From the run-level data frame (one row per run), the boolean column
"success_bool" (True when status == "success") is aggregated per config_label.
A 95 % Wilson score confidence interval is computed around each proportion to
account for the binomial nature of the outcome and the modest sample size
(n = 60 per config).

Display
-------
A grouped bar chart with one bar per configuration.  Error bars show the
Wilson CI.  Percentage labels sit above each bar.

Motivation
----------
This is the primary outcome figure.  It gives an immediate, at-a-glance
ranking of every configuration and makes it visually clear which model /
prompt / few-shot combinations succeed or fail.  The confidence intervals
quantify sampling uncertainty, preventing over-interpretation of small
differences (e.g., C1 at 13 % vs. C3 at 12 %).  It answers the headline
question: "Which configurations reliably produce compilable DSL?"


================================================================================
  Figure 2 — Main-Effect Forest Plot
  File: Report/Figures/fig02_main_effect_forest.png
================================================================================

Metric
------
Marginal success rate for each level of each factor (model, repair prompt,
few-shot examples), pooled across all other factors.

How the data is produced
------------------------
For every (factor, level) pair the run-level data is filtered and the success
rate computed.  A Wilson 95 % CI is attached.  A dashed vertical line marks the
overall (pooled) success rate across all 480 runs.

Display
-------
Forest-style dot-and-whisker plot.  Each row is a factor level, colour-coded
by factor group.

Motivation
----------
While Figure 1 shows all 8 combined configurations, Figure 2 isolates the
independent contribution of each factor.  It answers: "All else being equal,
does model choice matter more than prompt choice or few-shot provision?"
This is essential for a factorial experiment: it disentangles the main effects
before looking at interactions (Figure 3).  A reader can immediately see, for
instance, that the model factor has the largest spread, suggesting it is the
dominant driver of success.


================================================================================
  Figure 3 — Factor Interaction: Model × Few-Shot × Repair Prompt
  File: Report/Figures/fig03_factor_interaction.png
================================================================================

Metric
------
Success rate for each cell of the full 2 × 2 × 2 factorial design (model ×
few-shot × repair prompt).

How the data is produced
------------------------
The run-level data is sliced by repair prompt (one panel per SPR), then within
each panel by model (x-axis) and few-shot setting (separate lines).  Success
rate is computed for each cell.

Display
-------
Two-panel line plot (one per repair prompt).  Each line represents a few-shot
setting; the x-axis shows the two models.  Percentage annotations label each
point.

Motivation
----------
Main effects (Figure 2) can be misleading if interactions exist.  Figure 3
reveals whether the benefit of few-shot examples depends on the model or
prompt.  For example, if few-shot examples help Flash but are unnecessary for
Pro, that is an interaction not visible in the marginal averages.  This figure
is central to the comparative design: it shows whether the factors are additive
or synergistic.


================================================================================
  Figure 4 — Prompt × Scenario Success-Rate Heatmap
  File: Report/Figures/fig04_prompt_scenario_heatmap.png
================================================================================

Metric
------
Success rate for each (generative system prompt, scenario) pair, pooled across
all configurations.

How the data is produced
------------------------
A pivot table is built with system_prompt as rows and scenario as columns.
Each cell is the mean of success_bool for all runs sharing that prompt–scenario
pair (n = 8 per cell: 8 configs × 1 run).

Display
-------
Annotated heatmap (RdYlGn colour map, 0 %–100 %).

Motivation
----------
The study uses 5 generative system prompts and 4 scenarios.  This figure
reveals which prompt–scenario pairings are inherently easy or hard, independent
of configuration.  It is important for two reasons: (1) it highlights whether
some prompts generalise better across scenarios, and (2) it checks for
ceiling/floor effects — if a particular scenario is always 100 % or always 0 %,
that limits what the repair loop can tell us about model capability.


================================================================================
  Figure 5 — Iterations to Success (Pro-Model Configs)
  File: Report/Figures/fig05_iterations_box_strip.png
================================================================================

Metric
------
First-success iteration: the earliest repair iteration at which a run achieved
compiler_error_score == 0 (i.e., "derived_first_success_iteration").

How the data is produced
------------------------
Only successful runs from Pro-model configurations (C5–C8) are included.
Flash-model configs (C1–C4) are excluded because their very low success rates
(11–22 %) yield too few data points for a meaningful distribution.  The
derived_first_success_iteration column is extracted for each qualifying run.

Display
-------
Box-and-strip plot.  Box-plots show the median, IQR, and whiskers.  Individual
data points are jittered alongside.  Labels show (n_success / n_total) per
config.

Motivation
----------
Success rate alone does not capture efficiency: a configuration that always
succeeds on iteration 1 is more cost-effective than one that needs 5 rounds.
This figure quantifies how quickly the repair loop converges for the configs
that do succeed frequently (Pro-model).  It helps answer: "Among the
high-performing configurations, is there a meaningful difference in how fast
they reach a correct solution?"  The strip layer ensures outliers and
distributional shape are visible, not hidden inside a box.


================================================================================
  Figure 6 — Error-Score Convergence by Configuration
  File: Report/Figures/fig06_error_convergence.png
================================================================================

Metric
------
Median compiler error score at each repair iteration (1–10), per
configuration.

How the data is produced
------------------------
The iteration-level data frame (one row per (run, iteration)) is grouped by
(config_label, iteration).  The median of compiler_error_score is computed per
group.  Note: sample size decreases at later iterations because successful runs
exit the loop early, so later points reflect progressively harder (surviving)
runs.

Display
-------
Multi-line plot with one curve per configuration (C1–C8), coloured by
the per-config palette.

Motivation
----------
This figure visualises the trajectory of the repair process over time.  A
steeply declining curve indicates rapid error reduction; a flat or rising
curve indicates the model is stuck or regressing.  Comparing curves across
configs reveals whether certain setups (e.g., Pro + few-shot) converge faster.
The survivor-bias effect at later iterations is itself informative: if a
config's curve stays low, the few remaining runs are still making progress,
whereas a curve that climbs betrays regression in the hardest cases.


================================================================================
  Figure 7 — Error-Category Flow: Generation → Post-Repair (failed runs)
  File: Report/Figures/fig07_error_flow.png
================================================================================

Metric
------
For each of 11 compiler-error categories, the share (%) of total error
instances attributed to that category, measured at two points:

  1. Initial (generation): the error profile of the LLM's very first DSL
     output, before any repair iteration.
  2. Final (post-repair): the error profile after the last repair iteration
     (iteration 10 or the last attempted iteration).

Only failed runs are included (n = 241), since successful runs have zero
errors at termination.  A delta (Δ, in percentage-points) is annotated for
each category to quantify the net change.

How the data is produced
------------------------
From the iteration-level data frame, the first and last rows per run are
extracted.  For each error category, the total count of errors across all
failed runs is summed separately for first and last iterations.  Each
category's share = (category total / grand total) × 100.  The delta is
final_share − initial_share in percentage-points.  Categories with < 0.3 %
share at both stages are filtered out for readability.

Error categories tracked:

  Figure Label          Internal Key             Compiler Error Message
  ──────────────────    ─────────────────────    ──────────────────────────────────────
  Syntax Structure      err_syntax_structure     "required (...)+ loop did not match"
  Token Recognition     err_token_recognition    "token recognition error"
  Mismatched Input      err_mismatched_input     "mismatched input ... expecting"
  Extraneous Input      err_extraneous_input     "extraneous input ... expecting"
  Missing Token         err_missing_token        "missing ... at"
  Invalid Agent         err_semantic_agent       "Agent name must be one of"
  Invalid Resource      err_semantic_resource    "Resource name must be one of"
  Invalid Target        err_semantic_target      "Target name must be one of"
  Duplicate Agent       err_semantic_dup_agent   "same agent cannot be used"
  Ordering Violation    err_semantic_ordering    "value ... must be greater"
  Other                 err_other                Any [ERROR] line not matching above

Display
-------
Horizontal paired bar chart.  For each error category, a blue bar shows the
initial (generation) share and a red bar shows the final (post-repair) share.
A colour-coded delta badge (green = reduced, red = increased, grey = stable)
appears to the right of each pair.  The subtitle reports the total number of
failed runs and the raw error counts at each stage.

Motivation
----------
The previous error-category heatmap showed which runs had errors but not what
changed through the repair process.  This figure directly answers three
critical questions for the comparative study:

  1. What did the model fix?  Categories whose share drops (negative Δ) —
     e.g., mismatched_input and extraneous_input shrink, showing the repair
     loop resolves some parsing errors.

  2. What could the model not fix?  Categories whose share barely moves —
     e.g., semantic_agent stays dominant at ~50 %, indicating the model
     consistently fails to resolve agent-naming constraints.

  3. What did the model introduce?  Categories whose share rises (positive Δ)
     — e.g., semantic_resource and semantic_target increase, suggesting repair
     attempts sometimes create new semantic errors while trying to fix others
     (error cascading).

By controlling for total error volume (showing shares rather than raw counts),
the figure isolates compositional shifts: the overall error count drops 30 %
(22,604 → 15,819), but the mix of remaining errors shifts toward harder
semantic categories.  This is the key insight for future work: the repair loop
handles syntax/parsing effectively but struggles with semantic constraints,
pointing toward constrained decoding or agent-name grounding as the next
research direction.


================================================================================
  Figure 8 — Scenario Difficulty Profile
  File: Report/Figures/fig08_scenario_difficulty.png
================================================================================

Metric
------
Three metrics profiling each of the four scenarios:

  Panel A – Failure Rate:
    1 − success_rate, pooled across all 120 runs per scenario (8 configs ×
    5 system prompts × 3 runs = 120 per scenario).

  Panel B – Mean Final Error Score (failed runs only):
    The average of derived_final_error_score among runs that never reached
    compiler_error_score == 0.  This quantifies how far from success the
    failed attempts ended.

  Panel C – Mean Initial Error Score (all runs):
    The average of derived_initial_error_score across all runs per scenario,
    regardless of outcome.  This quantifies how many compiler errors the
    LLM's first attempt produces — a proxy for the scenario's inherent
    generative difficulty before any repair is attempted.

How the data is produced
------------------------
The run-level data frame is grouped by scenario.  Panel A computes
1 − mean(success_bool).  Panel B filters to failed runs and averages
derived_final_error_score.  Panel C averages derived_initial_error_score
over all runs.

Display
-------
Three side-by-side bar charts sharing the same x-axis (scenarios).  Each panel
uses a distinct colour (red, orange, blue) for visual separation.

Motivation
----------
Not all scenarios are equally difficult, and understanding the source of that
difficulty is important for interpreting the other figures.  Panel A gives the
raw difficulty ranking.  Panel B shows the residual severity: a scenario could
have a high failure rate but low final error (close to success), or a high
failure rate *and* high residual error (far from success) — these have
different implications for the repair loop's potential.  Panel C (mean initial
error score) captures the scenario's generative complexity: a high initial
score means the LLM's very first attempt already contains many errors, placing
a heavier burden on the repair loop.  Comparing Panel C with Panel A reveals
whether scenarios that start badly also finish badly, or whether the repair
loop can recover from a rough start.  Together the three panels give a concise
"difficulty fingerprint" per scenario.
